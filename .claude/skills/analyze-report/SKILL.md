## 목적

현재 프로젝트의 Google Sheets 두 시트에서 미분석 신규 데이터를 읽어 **프로젝트 루트 및 하위 모듈의 `CLAUDE.md` / `CLAUDE.local.md`** 와 비교 분석한 뒤 규칙 개선안을 도출한다.

| 소스 | 식별 | 성격 |
|------|------|------|
| 회고 데이터 | `GOOGLE_SHEETS_SESSION_LOG_GID` | 자유 서술 — 잘된점·잘못된점·원인·개선 |
| 개인 규칙 후보 | `GOOGLE_SHEETS_PERSONAL_RULES_GID` | 정제된 규칙 — `share-rules` 가 각자 `CLAUDE.local.md` 공유 가능 섹션에서 수집 |

두 소스는 보완적이다. 동일 마찰을 두 소스에서 모두 짚고 있으면 강한 승격 후보다.

> 본 스킬은 **각 프로젝트 루트에서 실행**되는 것을 전제로 한다. 시트는 프로젝트별로 분리되어 있으므로 한 번 실행 = 그 프로젝트 한 건의 분석이다.

## 1단계: 환경변수 확인

아래 환경변수가 설정되어 있는지 확인한다.

| 환경변수 | 설명 |
|----------|------|
| `PROJECT_NAME` | 보고서에 표시할 프로젝트명 (디렉토리명과 무관하게 고정) |
| `GOOGLE_SERVICE_ACCOUNT_KEY_PATH` | 서비스 계정 JSON 키 파일 경로 |
| `GOOGLE_SHEETS_FEEDBACK_ID` | Google Spreadsheet ID |
| `GOOGLE_SHEETS_SESSION_LOG_GID` | 회고 데이터 시트의 gid |
| `GOOGLE_SHEETS_PERSONAL_RULES_GID` | 개인 규칙 후보 시트의 gid |

없으면 `.claude/guides/google-sheets-setup.md` 를 참고하여 설정을 안내하고 중단한다.

---

## 2단계: 시트 데이터 조회

아래 두 스크립트를 실행하여 각 시트의 미분석 행을 가져온다.

```bash
node "{SKILL_DIR}/scripts/fetch-sheet.js"
node "{SKILL_DIR}/scripts/fetch-personal-rules.js"
```

> `SKILL_DIR` 은 스킬 실행 시 상단에 표시되는 Base directory 경로다.

각 스크립트는 "클로드 분석여부" 컬럼이 비어있는 행만 필터링하여 JSON 배열로 출력한다.
`개인 규칙 후보` 시트가 아직 만들어지지 않은 경우 빈 배열을 반환한다.

두 소스 모두 신규 행이 없으면 아래 메시지를 출력하고 종료한다.

```
분석할 새 데이터가 없습니다.
```

## 3단계: 현재 규칙 파일 수집

cwd(프로젝트 루트)의 모든 `CLAUDE.md` / `CLAUDE.local.md` 를 글로브로 수집해 "현재 적용 중인 규칙" 컨텍스트를 만든다.

```
./CLAUDE.md
./CLAUDE.local.md
./**/CLAUDE.md             ← 하위 모듈 규칙
./**/CLAUDE.local.md       ← 하위 모듈 개인 규칙
```

> 다음 폴더는 제외: `node_modules`, `dist`, `build`, `target`, `.git`, `.claude/skills`, `.claude/hooks`, `reports`.

수집된 파일들의 경로 목록과 본문을 분석에 함께 입력한다. 어느 파일이 어느 모듈 소속인지(경로) 가 분석 시 "적용 위치(추정)" 판단의 근거가 된다.

## 4단계: 분석

신규 행 데이터를 아래 관점으로 분석한다.

### 분석 기준

**회고 데이터**
1. **잘못된점** 중 현재 규칙으로 예방 가능했던 것 → 규칙 보완 필요
2. **잘못된점** 중 현재 규칙이 있는데 지켜지지 않은 것 → 규칙 표현 강화 필요
3. **잘못된점** 중 새로운 패턴 → 신규 규칙 추가 검토
4. **잘된점** 중 반복 성공 패턴 → 적용 중인 규칙 효과 확인
5. **개선내용 / 기타** 에 사용자가 직접 제안한 내용 → 그대로 반영 검토

**개인 규칙 후보**
6. 동일 또는 유사 규칙이 **여러 작성자(`이름`)에게서 독립 등장** → 강한 승격 후보 (루트 `CLAUDE.md`)
7. 한 명만 제안한 규칙 → 약한 후보 (보고서에 표시만, 즉시 승격 금지)
8. 회고(잘못된점) 패턴과 매칭되는 규칙 → 매우 강한 승격 후보 (두 소스에서 동시 검출)

### 분류 태그

| 태그 | 기준 |
|------|------|
| `[규칙-누락]` | 관련 규칙이 없어서 발생한 문제 |
| `[규칙-미적용]` | 규칙이 있는데 Claude가 따르지 않은 경우 |
| `[사용자-가이드]` | 규칙보다 사용자 프롬프트 방식으로 해결해야 할 것 |
| `[정보-부족]` | 규칙 추가보다 참조 정보(함수 목록, 매핑표 등) 보완이 필요한 것 |

### 규칙 타입 분류

모든 규칙 후보는 아래 5개 타입 중 하나로 분류한다. 사람이 빠르게 라우팅할 수 있도록 보고서 표에 `규칙 타입` 컬럼으로 표시한다.

| 타입 | 정의 | 기본 적용 위치 |
|------|------|---------------|
| `[코딩규칙]` | 코드 작성·수정 시 따라야 할 스타일·문법·네이밍 | `.claude/rules/{lang}-style.md` 또는 모듈 `CLAUDE.md` |
| `[탐색규칙]` | 코드 탐색·파일 위치·작업 순서 패턴 | 루트 `CLAUDE.md` > 코드 탐색 / `.claude/rules/{도메인}.md` |
| `[지식]` | 프로젝트 도메인·환경 정보 (DB vendor, properties 매핑, 인터셉터 패턴 등) | 해당 모듈 `CLAUDE.md` > 프로젝트 개요·필수 패턴 |
| `[워크플로우]` | Claude 의 작업 절차·승인 지점·금지 행동 (커밋·푸시·파일 수정 범위 등) | 루트 `CLAUDE.md` > AI 협업 지침 |
| `[사용자-가이드]` | 규칙이 아니라 사용자 프롬프트 방식으로 해결할 항목 (적용 대상 아님) | — (사용자 가이드 섹션에만 표시) |

분류가 모호한 경우 보수적으로 `[코딩규칙]` 또는 `[지식]`으로 분류한다.

### 적용 위치 추정

데이터에 모듈 정보가 없으므로 — 피드백 본문과 규칙 파일 내용을 비교해 **가장 어울리는 파일 경로**를 보고서에 "(추정)" 표시와 함께 제시한다. 확정은 사용자가 보고서를 보고 판단한다.

| 태그 | 규칙 타입 | 문제 패턴 | 출처 | 제안 내용 | 적용 위치(추정) |

> 적용 위치(추정)는 수집된 규칙 파일 중 가장 어울리는 경로 (예: `./CLAUDE.md`, `./modules/pce/CLAUDE.md`).

## 개인 규칙 후보 승격 검토

| 강도 | 규칙 타입 | 규칙 요약 | 작성자 수 | 회고 매칭 | 승격 대상 파일 |
|------|----------|-----------|-----------|-----------|---------------|

> 강도 기준: **강** = 2명 이상 + 회고 매칭 / **중** = 2명 이상 OR 회고 매칭 / **약** = 1명 단독
> 규칙 타입은 위 "규칙 타입 분류" 표의 5개 중 하나로 표기한다.

## 6단계: 사용자 확인 및 시트 업데이트

보고서 경로를 안내하고 아래 메시지를 출력한다.

```
보고서가 생성되었습니다: reports/analyze-report/{날짜}-analyze-report.md

분석한 {N}건의 데이터를 시트에 완료 표시할까요? (예/아니오)
```

사용자가 확인하면 아래 두 스크립트를 실행하여 각 시트의 해당 행에 오늘 날짜를 기입한다.

```bash
node "{SKILL_DIR}/scripts/mark-analyzed.js"            {회고_rowIndex1} {회고_rowIndex2} ...
node "{SKILL_DIR}/scripts/mark-personal-rules-analyzed.js" {규칙_rowIndex1} {규칙_rowIndex2} ...
```

> rowIndex는 2단계에서 조회한 각 행의 `rowIndex` 값을 공백으로 나열한다.
> 어느 한 쪽이 0건이면 해당 스크립트는 호출하지 않는다.
