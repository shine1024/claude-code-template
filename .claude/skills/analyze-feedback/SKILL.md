## 목적

Google Sheets 의 두 소스에서 미분석 신규 데이터를 읽어 현재 프로젝트 CLAUDE-TEMPLATE 파일들과 비교 분석 후 규칙 개선안을 도출합니다.

| 소스 | 식별 | 성격 |
|------|------|------|
| 회고 데이터 | `GOOGLE_SHEETS_SESSION_LOG_GID` | 자유 서술 — 잘된점·잘못된점·원인·개선 |
| 개인 규칙 후보 | `GOOGLE_SHEETS_PERSONAL_RULES_GID` | 정제된 규칙 — `share-rules` 가 각자 `CLAUDE.local.md` 공유 가능 섹션에서 수집 |

두 소스는 보완적입니다. 동일 마찰을 두 소스에서 모두 짚고 있으면 강한 승격 후보입니다.

---

## 1단계: 환경변수 확인

아래 환경변수가 설정되어 있는지 확인합니다.

| 환경변수 | 설명 |
|----------|------|
| `GOOGLE_SERVICE_ACCOUNT_KEY_PATH` | 서비스 계정 JSON 키 파일 경로 |
| `GOOGLE_SHEETS_FEEDBACK_ID` | Google Spreadsheet ID |
| `GOOGLE_SHEETS_SESSION_LOG_GID` | 회고 데이터 시트의 gid |
| `GOOGLE_SHEETS_PERSONAL_RULES_GID` | 개인 규칙 후보 시트의 gid |

없으면 `.claude/guides/google-sheets-setup.md` 를 참고하여 설정을 안내하고 중단합니다.

---

## 2단계: 시트 데이터 조회

아래 두 스크립트를 실행하여 각 시트의 미분석 행을 가져옵니다.

```bash
node "{SKILL_DIR}/scripts/fetch-sheet.js"
node "{SKILL_DIR}/scripts/fetch-personal-rules.js"
```

> `SKILL_DIR` 은 스킬 실행 시 상단에 표시되는 Base directory 경로입니다.

각 스크립트는 "클로드 분석여부" 컬럼이 비어있는 행만 필터링하여 JSON 배열로 출력합니다.
`개인 규칙 후보` 시트가 아직 만들어지지 않은 경우 빈 배열을 반환합니다.

두 소스 모두 신규 행이 없으면 아래 메시지를 출력하고 종료합니다.

```
분석할 새 데이터가 없습니다.
```

## 3단계: 현재 템플릿 규칙 읽기

신규 행에 등장하는 프로젝트에 해당하는 템플릿 파일을 읽어 현재 적용된 규칙을 파악합니다.

| 프로젝트 | 읽을 파일 |
|----------|-----------|
| UniFLOW 웹 | `template/CLAUDE-TEMPLATE-uniflow.md` |
| UniWORKS 웹 | `template/CLAUDE-TEMPLATE-uniworks-pce.md`, `template/CLAUDE-TEMPLATE-uniworks-public.md` |
| 공통 | `template/CLAUDE-TEMPLATE.md` |

## 4단계: 분석

신규 행 데이터를 아래 관점으로 분석합니다.

### 분석 기준

**`2) 작업세션 수집` (회고 데이터)**
1. **잘못된점** 중 현재 템플릿 규칙으로 예방 가능했던 것 → 규칙 보완 필요
2. **잘못된점** 중 현재 규칙이 있는데 지켜지지 않은 것 → 규칙 표현 강화 필요
3. **잘못된점** 중 새로운 패턴 → 신규 규칙 추가 검토
4. **잘된점** 중 반복 성공 패턴 → 적용 중인 규칙 효과 확인
5. **개선내용 / 기타** 에 사용자가 직접 제안한 내용 → 그대로 반영 검토

**`개인 규칙 후보` (개인 규칙)**
6. 동일 또는 유사 규칙이 **여러 작성자(`이름`)에게서 독립 등장** → 강한 승격 후보 (글로벌 CLAUDE.md 또는 `template/CLAUDE-TEMPLATE.md`)
7. 한 명만 제안한 규칙 → 약한 후보 (보고서에 표시만, 즉시 승격 금지)
8. 회고(잘못된점) 패턴과 매칭되는 규칙 → 매우 강한 승격 후보 (두 소스에서 동시 검출)

### 분류 태그

| 태그 | 기준 |
|------|------|
| `[규칙-누락]` | 관련 규칙이 없어서 발생한 문제 |
| `[규칙-미적용]` | 규칙이 있는데 Claude가 따르지 않은 경우 |
| `[사용자-가이드]` | 규칙보다 사용자 프롬프트 방식으로 해결해야 할 것 |
| `[정보-부족]` | 규칙 추가보다 참조 정보(함수 목록, 매핑표 등) 보완이 필요한 것 |

## 6단계: 사용자 확인 및 시트 업데이트

보고서 경로를 안내하고 아래 메시지를 출력합니다.

```
보고서가 생성되었습니다: reports/feedback-analysis-{날짜}.md

분석한 {N}건의 데이터를 시트에 완료 표시할까요? (예/아니오)
```

사용자가 확인하면 아래 두 스크립트를 실행하여 각 시트의 해당 행에 오늘 날짜를 기입합니다.

```bash
node "{SKILL_DIR}/scripts/mark-analyzed.js"            {회고_rowIndex1} {회고_rowIndex2} ...
node "{SKILL_DIR}/scripts/mark-personal-rules-analyzed.js" {규칙_rowIndex1} {규칙_rowIndex2} ...
```

> rowIndex는 2단계에서 조회한 각 행의 `rowIndex` 값을 공백으로 나열합니다.
> 어느 한 쪽이 0건이면 해당 스크립트는 호출하지 않습니다.
