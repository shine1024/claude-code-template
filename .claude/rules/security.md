# 보안 규칙

## 시크릿 처리
- API 키·비밀번호·토큰 등 시크릿은 코드에 하드코딩 금지
- 환경변수 또는 시크릿 매니저로 주입
- 시크릿이 코드·로그·커밋에 노출됐다면 즉시 폐기·교체 후 작업 재개
- `.env`, 자격증명 파일은 반드시 `.gitignore` 대상

## 입력 검증
- 외부에서 들어오는 데이터(API 요청, 사용자 입력, 파일, 외부 시스템 응답)는 시스템 경계에서 검증한다
- 내부 함수 간 호출에서는 중복 검증하지 않는다
- 검증 실패 시 안전한 기본값 또는 명시적 오류로 처리

## SQL 인젝션 방지
- 동적 SQL 문자열 결합 금지 → 파라미터 바인딩 사용
- MyBatis: `${}` 대신 `#{}` 사용
  - `${}`는 문자열 치환이라 인젝션에 취약. 정렬 컬럼·테이블명 등 바인딩 불가능한 경우에만 화이트리스트 검증 후 사용
- 사용자 입력을 LIKE 절에 넣을 때 `%`, `_` 이스케이프

## XSS 방지
- 사용자 입력을 HTML로 출력할 때 반드시 이스케이프
- `innerHTML`, `document.write` 등 직접 삽입 함수 지양 (불가피한 경우 새니타이즈 라이브러리 적용)
- URL을 동적으로 생성할 때 `encodeURIComponent` 적용

## 에러 응답
- 클라이언트에 stack trace, SQL 쿼리, 내부 파일 경로 노출 금지
- 인증·인가 실패는 동일한 응답으로 통일 (계정 존재 여부 누설 금지)

## 인시던트 발견 시
- 보안 문제 발견 시 작업 중단하고 사용자에게 보고
- 임의로 우회하지 말 것 (예: `--no-verify`, 권한 완화, 검증 로직 비활성화)

## 체크리스트
- [ ] 코드·커밋에 시크릿이 포함되지 않았는가
- [ ] 외부 입력은 경계에서 검증되는가
- [ ] DB 쿼리는 파라미터 바인딩을 사용하는가
- [ ] HTML 출력에 이스케이프가 적용되는가
- [ ] 에러 응답에 민감 정보가 노출되지 않는가
