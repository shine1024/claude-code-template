# 성능 규칙

코드 작성 시 일반적으로 적용하는 성능 원칙.

## DB 접근
- N+1 쿼리 회피 — 반복문 안에서 단건 SELECT를 호출하지 말고 `IN` 절·`JOIN`으로 한 번에 조회
- WHERE·JOIN 조건 컬럼이 인덱스를 탈 수 있는지 확인
  - 컬럼에 함수 적용(`UPPER(col) = ...`), 묵시적 형변환은 인덱스 사용을 막는다
- 필요한 컬럼만 SELECT — `SELECT *` 지양 (특히 페이지에 표시할 목록 쿼리)
- 큰 테이블의 페이징은 `LIMIT/OFFSET` 외에 인덱스 기반 keyset 페이징 가능 여부 검토

## 반복·컬렉션
- 반복문 안에서 동일한 결과를 반복 계산·재조회하지 않는다 → 루프 밖에서 한 번만 수행
- 조회 결과가 클 가능성이 있으면 스트리밍·페이지 단위 처리, 메모리 한 번에 적재 금지
- 잦은 검색이 필요한 데이터는 List 대신 Map·Set으로 인덱싱

## 캐싱
- 자주 호출되고 결과 변동이 적은 데이터만 캐싱 대상으로 검토
- 캐시 키·만료 시점·무효화 시점을 명확히 정의하지 않은 캐싱은 도입하지 않는다

## 외부 호출
- 외부 API·DB 호출은 타임아웃을 명시한다 (무한 대기 금지)
- 가능하면 배치(bulk) API 사용, 호출 횟수를 줄인다

## 측정 우선
- 추측으로 최적화하지 말 것 — 쿼리 plan, 프로파일링, 로그로 병목을 먼저 확인한다
- 가독성과 트레이드오프가 발생하는 최적화는 실측 결과로 정당화

## 체크리스트
- [ ] 반복문 안에 DB·외부 API 호출이 있는가 (있다면 합칠 수 있는가)
- [ ] 추가한 쿼리가 인덱스를 사용하는가
- [ ] 큰 결과셋을 한 번에 메모리에 적재하지 않는가
- [ ] 외부 호출에 타임아웃이 설정되어 있는가
