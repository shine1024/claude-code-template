## 1. Claude Code란?

Claude Code는 Anthropic이 만든 **에이전트 코딩 도구**입니다. 단순히 질문에 답하는 챗봇이 아니라, 실제로 파일을 읽고, 코드를 수정하고, 명령을 실행하며, 문제를 자율적으로 해결하는 AI 개발 파트너입니다.

```
Claude Code = AI 두뇌 + 파일 읽기/편집 + 명령 실행 + Git 연동 + 외부 도구 연결
```

**핵심 특징:**
- 코드베이스 전체를 이해하고 여러 파일에 걸친 작업 수행
- 파일 읽기/편집, 터미널 명령 실행, Git 작업 자동화
- 자율적으로 계획하고, 구현하고, 검증
- 여러분이 자리를 비운 동안에도 작업 수행 가능

### 사용 가능한 환경

| 환경 | 특징 |
|------|------|
| **터미널 CLI** | 가장 완전한 기능, 파이프/스크립트 연동 |
| **VS Code 확장** | 인라인 diff, @-멘션, Plan 검토 |
| **JetBrains 플러그인** | IntelliJ IDEA, WebStorm 등 지원 |
| **데스크톱 앱** | 시각적 diff, 병렬 세션, 예약 작업 |
| **웹 (claude.ai/code)** | 브라우저에서 로컬 설치 없이 사용 |

## 2. 설치 및 시작하기

### 설치

**macOS / Linux**
```bash
npm install -g @anthropic-ai/claude-code
```

**Windows (winget — 권장)**
```powershell
winget install Anthropic.ClaudeCode
```

> Windows는 [Git for Windows](https://git-scm.com/downloads/win) 사전 설치 필요

> **winget을 쓸 수 없는 환경에서만** 보조 방식으로 PowerShell 스크립트 사용:
> ```powershell
> irm https://claude.ai/install.ps1 | iex
> ```
> PowerShell 실행 정책·TLS 설정에 따라 실패할 수 있습니다. 이 경우 `Set-ExecutionPolicy -Scope CurrentUser RemoteSigned` 등 추가 설정이 필요할 수 있습니다.

### VS Code 확장 설치 (권장)

1. VS Code 확장 탭 열기 (`Ctrl+Shift+X`)
2. `Claude Code` 검색 후 설치
3. 사이드바 또는 `Ctrl+Shift+P` → `Claude Code: Open` 으로 실행

### JetBrains 플러그인 설치

1. Settings → Plugins → Marketplace
2. `Claude Code` 검색 후 설치
3. 우측 사이드바 Claude 아이콘으로 실행

### 첫 실행

```bash
# 프로젝트 디렉토리로 이동
cd /path/to/your/project

# Claude Code 시작 (첫 실행 시 로그인 진행)
claude
```

## 5. 자주 헷갈리는 명령어·모드 비교

처음 쓰면 이름이 비슷해서 헷갈리는 짝들입니다. 무엇을 언제 써야 하는지부터 잡아두면 워크플로우가 빨라집니다.

### 컨텍스트 정리: `/clear` vs `/compact`

| 명령 | 동작 | 언제 쓰나 |
|------|------|----------|
| `/clear` | 대화 컨텍스트를 **완전히 비움** | 관련 없는 다른 작업으로 전환할 때 |
| `/compact` | 대화를 **요약 압축** (핵심 결정·진행 상황 유지) | 같은 작업을 이어가지만 컨텍스트가 길어졌을 때 |

> 판단 기준: 이전 대화 내용이 **다음 작업에도 쓸모 있는가?** 있으면 compact, 없으면 clear.

### 세션 재개: `claude -c` vs `claude -r` vs `claude -p`

| 옵션 | 동작 |
|------|------|
| `claude -c` | 가장 **최근 세션을 자동으로** 이어서 시작 |
| `claude -r` | 세션 **목록 UI에서 골라서** 재개 |
| `claude -p "..."` | 한 번 응답하고 **즉시 종료** (스크립트·파이프 연동용) |

### 작업 중단: `Esc` vs `Esc+Esc` vs `Ctrl+C`

| 키 | 동작 |
|------|------|
| `Esc` | 진행 중인 작업만 중단, **컨텍스트는 유지** |
| `Esc+Esc` | **Rewind 메뉴** 열기 — 이전 상태로 되돌리기 |
| `Ctrl+C` | 작업 중단 (한 번 더 누르면 세션 종료) |

> Rewind는 Claude의 파일 변경·메시지를 되돌리지만 **이미 실행된 shell 명령은 되돌릴 수 없습니다.**

### 파일·폴더 참조: `@`

프롬프트에서 파일·폴더를 직접 컨텍스트로 가져오는 문법입니다. 한 프롬프트에 여러 개를 같이 써도 됩니다.

```
@src/service/OrderService.java 이 서비스의 책임을 정리해줘.
@src/                            반복되는 패턴을 한 줄씩 요약해줘.
```

> 제약 조건(예: "인터페이스 변경 금지", "다른 파일 건드리지 말 것")은 별도 문법 없이 자연어 문장으로 그대로 적습니다.

### 작동 모드: Normal vs Auto vs Plan

`Shift+Tab`을 반복해서 전환합니다.

| 모드 | 파일 수정 | 권한 확인 | 언제 쓰나 |
|------|---------|---------|----------|
| **Normal** | 가능 | 매번 확인 | 신중하게 작업하고 싶을 때 (기본) |
| **Auto** | 가능 | 사전 허용된 명령은 자동 실행 | 익숙한 반복 작업, 흐름이 끊기는 게 싫을 때 |
| **Plan** | **불가** (읽기 전용) | — | 분석·계획만 수행. 큰 작업의 사전 검토용 |

> 모델·노력 수준(opus/sonnet/haiku, `/effort`) 비교는 [심화 가이드 §1](./02_claude-code_심화.md#1-plan-mode-심화-활용) 참고.

## 6. 핵심 워크플로우

워크플로우 4종(탐색·버그 수정·기능 개발·커밋)을 흐름 단위로 익힙니다. `@` 문법은 [§5](#5-자주-헷갈리는-명령어모드-비교)를 참고하세요.

### 코드베이스 탐색

처음에 전체 맥락을 잡는 것이 중요합니다:

```
이 프로젝트가 어떤 구조로 되어 있는지 설명해줘.
```
```
주요 데이터 흐름을 프론트엔드부터 백엔드까지 설명해줘.
```
```
@src/main/java/com/example/service/OrderService.java 이 서비스의 역할을 설명해줘.
```

---

### 버그 수정

```
로그인 처리 중 세션이 유지되지 않는 버그가 있어.
에러 메시지: [에러 내용 붙여넣기]
@src/main/java/com/example/controller/AuthController.java 와
@src/main/webapp/js/login.js 를 확인하고 원인을 찾아줘.
```

**증상이 아닌 근본 원인을 설명하세요:**

| 모호한 요청 | 구체적인 요청 |
|------------|-------------|
| "그리드 버그 수정해줘" | "order.js에서 수량 수정 시 합계가 갱신되지 않음. onCellEdit 핸들러 확인 후 수정해줘." |

**에러 로그를 직접 파이프할 수도 있습니다:**
```bash
cat error.log | claude -p "이 에러의 근본 원인을 분석해줘."
```

### 새 기능 개발

**좋은 프롬프트의 4요소** — 새 기능 요청은 이 네 가지를 채울수록 결과가 좋아집니다.

| 요소 | 설명 | 빠지면 |
|------|------|--------|
| **대상** | 어떤 파일·함수·요소를 건드리는지 | Claude가 엉뚱한 곳을 고침 |
| **동작** | 어떤 입력에서 어떤 결과가 나와야 하는지 | 동작이 의도와 어긋남 |
| **제약** | 변경 금지 항목, 사용할/사용하지 말 라이브러리 | 사이드이펙트 발생 |
| **검증** | 어떻게 확인할 수 있는지 (빌드·테스트·시나리오) | 끝까지 가지 않고 중단 |

**모호한 요청 vs 구체적인 요청**

```
[Before — 모호함]
@src/main/webapp/js/order.js
저장 버튼 클릭 시 확인 다이얼로그를 추가해줘.
```

```
[After — 4요소가 들어감]
@src/main/webapp/js/order.js
저장 버튼(#btn-save) 클릭 시 표준 confirm() 으로 확인 다이얼로그를 띄우고,
"확인"을 누른 경우에만 기존 saveOrder() 를 호출하도록 수정해줘.

조건:
- 메시지: "주문을 저장하시겠습니까?"
- saveOrder() 함수 시그니처는 변경 금지
- 커스텀 모달 도입 금지 (표준 confirm 만 사용)
- 변경 후 어떤 시나리오로 브라우저 테스트하면 되는지 안내까지 포함
```

**중규모 기능 (Plan Mode 활용 권장):**
```
[Plan Mode로 전환 후]
주문 목록에 엑셀 다운로드 기능을 추가하려고 해.

먼저 현재 코드를 파악해줘:
- OrderController.java 의 목록 조회 API 시그니처
- order.js 의 데이터 바인딩 구조
- 같은 도메인에서 이미 쓰고 있는 엑셀 라이브러리(있다면)

그 다음 구현 계획을 세워줘:
- 백엔드 엔드포인트 추가 vs 기존 응답 가공
- 라이브러리 선택 (이미 있는 것 우선)
- 권한 체크 누락 방지 방안
```

**Plan Mode 전환:** `Shift+Tab` 반복 → `⏸ plan mode on` 표시될 때까지

## 7. CLAUDE.md 기본 설정

### CLAUDE.md란?

Claude Code가 **매 세션 시작 시 자동으로 읽는** 마크다운 파일입니다. 프로젝트 규칙, 코딩 표준, 빌드 명령 등을 작성해두면 Claude가 항상 그 맥락을 가지고 작업합니다.

```
CLAUDE.md = Claude에게 보내는 지속적인 지시서
```

### `/init` 으로 자동 생성

```bash
# 프로젝트 루트에서 실행하면 코드베이스를 분석해서 CLAUDE.md 초안 생성
/init
```

### 파일 배치 위치

| 범위 | 위치 | Git 공유 |
|------|------|----------|
| **전역 (모든 프로젝트)** | 사용자 홈 폴더의 `.claude/CLAUDE.md`<br>· Windows: `C:\Users\<사용자명>\.claude\CLAUDE.md`<br>· macOS·Linux: `~/.claude/CLAUDE.md` | ❌ 개인 유지 |
| **프로젝트 (팀 공유)** | 프로젝트 루트의 `CLAUDE.md` | ✅ 커밋 권장 |
| **프로젝트 (개인)** | 프로젝트 루트의 `CLAUDE.local.md` | ❌ gitignore 추가 |

### CLAUDE.md 작성 핵심 원칙

- **200줄 이하** 유지 (길수록 준수율 저하)
- **구체적으로** 작성 ("코드 정리" ❌ → "DTO/VO 클래스 사용 필수" ✅)
- **자주 검토**하여 오래된/충돌하는 지침 제거

### 예시

```markdown
## 빌드 및 실행
- 빌드: `./gradlew build`
- 서버 실행: `./gradlew bootRun`
- 테스트: `./gradlew test`

## 코딩 규칙
- 외부 API 호출 결과는 반드시 DTO로 받아 변환 후 사용
- DB 트랜잭션: 서비스 레이어에서만 @Transactional 선언
- 커밋 메시지 형식: `refs #이슈번호 [모듈명] - [변경 내용]`

## 주의사항
- .env.production 파일은 절대 수정하지 말 것
- 외부 API 응답 파싱 시 null 체크 필수
```

## 8. 핵심 모범 사례

### 검증 방법을 함께 제공하라

Claude가 스스로 검증할 수 있을 때 결과 품질이 향상됩니다:

```
getOrderList API를 호출해서 목록 반환하도록 수정해줘.
빌드 후 서버 기동해서 실제로 확인해줘.
```

### 탐색 → 계획 → 구현 순서를 지켜라

```
잘못된 방식: "기능 추가해줘" → 즉시 코드 작성
올바른 방식: 탐색(Plan Mode) → 계획 검토 → 구현
```

### 컨텍스트를 초기화하며 작업하라

관련 없는 작업 사이에는 반드시 컨텍스트를 정리합니다. **세 가지 옵션** 중 작업 단위에 맞춰 고릅니다.

| 옵션 | 동작 | 언제 |
|------|------|------|
| `/compact` | 중요한 내용을 유지한 채 **압축** | 같은 작업을 이어가는데 컨텍스트가 길어졌을 때 |
| `/clear` | 컨텍스트 **완전 초기화** (세션은 유지) | 같은 세션에서 다른 작업으로 전환할 때 |
| **세션 종료 후 새로 시작** | 가장 깨끗한 상태 — 누적 영향 0 | 큰 작업 단위가 하나 끝났을 때 (가장 권장) |

```bash
/compact "API 연동 변경사항만 유지"   # 압축 시 유지할 핵심 명시
/clear
exit                                  # 또는 Ctrl+D — 세션 종료 후 다시 claude 실행
```

> 큰 작업이 끝났는데 같은 세션을 계속 끌고 가면, 다음 작업에 의도치 않은 영향(이전 결정·실패 시도 등)이 남기 쉽습니다. 작업 단위 사이의 **세션 종료 → 재시작**이 가장 안전한 분리 방법입니다.

### 작업 규모에 맞게 접근하라

| 규모 | 접근 방식 |
|------|---------|
| 오타 수정, 로그 추가 | Plan Mode 없이 바로 진행 |
| 단일 기능 수정 | Plan Mode로 계획 후 구현 |
| 신규 기능, 리팩토링 | 탐색 → 계획 검토 → 단계적 구현 |

## 9. 실습: 30분 첫 작업

기초 개념을 손에 익히는 시간입니다. 부담 없이 따라 할 수 있도록 **새 실습용 폴더**를 하나 만들어 그 안에서 진행합니다. 본인의 실제 프로젝트는 건드리지 않습니다.

> **실습 과제**: 빈 폴더에서 시작 → Claude로 작은 TODO 리스트 앱 생성 → 코드 탐색 → CLAUDE.md 작성 → Plan Mode로 "완료 항목 일괄 삭제" 기능 추가 → 브라우저 검증 → 커밋. **vanilla HTML/CSS/JS** 만 사용합니다.

### 0. 준비 (1분)

새 실습용 폴더 생성:

```bash
mkdir todo-practice
cd todo-practice
git init
```

### 1. 첫 실행 + 초기 코드 생성 (5분)

```bash
claude
```

세션이 열리면 첫 요청으로 앱 골격을 만들게 합니다.

```
간단한 TODO 리스트 웹앱을 만들어줘.

요구사항:
- index.html, style.css, app.js 3개 파일로 구성
- 프레임워크 없이 vanilla JavaScript 만 사용
- 기능: 항목 추가 / 완료 토글(체크박스) / 항목 삭제
- 데이터는 localStorage 에 저장 (새로고침 후에도 유지)
- 디자인은 깔끔한 단일 컬럼 레이아웃, 모바일에서도 보기 좋게
- 변경 후 브라우저에서 어떻게 열어 확인하는지도 알려줘
```

생성이 끝나면 안내대로 `index.html` 을 브라우저에서 열어 동작을 한 번 확인합니다 — 항목 추가, 완료 토글, 삭제, 새로고침 후 유지까지.

### 2. 코드베이스 탐색 (3분)

생성된 코드를 Claude의 시각으로 한 번 정리해 봅니다.

```
방금 만든 TODO 앱을 파일별로 한 줄씩 정리해줘.
주요 함수와 그 역할도 함께.
어떤 함수가 localStorage 와 직접 접촉하는지도 알려줘.
```

응답을 그대로 받아들이지 말고, **한 파일이라도 직접 열어** Claude의 설명과 일치하는지 점검합니다. 어긋나는 부분이 있으면 거기서부터 좁혀 묻습니다.

### 3. CLAUDE.md 초기화 (3분)

```
/init
```

자동 생성된 `CLAUDE.md` 를 한 번 훑어보고, 다음 한 줄을 직접 추가합니다 (편집기로 직접 넣어도 되고, Claude Code 에게 "CLAUDE.md 에 ~ 규칙 추가해줘" 라고 지시해도 됩니다).

```markdown
## 프로젝트 규칙
- 외부 라이브러리·프레임워크 도입 금지 (vanilla JS 유지)
```

> 1단계에서 "vanilla JS만" 이라고 요구했지만 자동 생성된 CLAUDE.md 에 그 제약이 빠져 있다면, 4단계 Plan Mode 에서 Claude가 멋대로 라이브러리를 끌어올 수 있습니다. **요청에서 한 번 말한 제약을 CLAUDE.md 로 영구화하는 것**이 이 단계의 핵심입니다.

> CLAUDE.md 는 길수록 Claude가 덜 따릅니다. 200줄 이하 원칙.

### 4. Plan Mode로 기능 추가 계획 (10분)

`Shift+Tab` 을 두 번 눌러 **Plan Mode**로 전환 (`⏸ plan mode on` 표시).

추가할 기능: **"완료된 항목 한 번에 삭제" 버튼** — 4요소(대상·동작·제약·검증)를 갖춘 프롬프트를 직접 써 봅니다.

```
@app.js @index.html @style.css
완료된 TODO 항목을 한 번에 삭제하는 "완료 항목 비우기" 버튼을 추가하려고 해.
구현하지는 말고 계획만 세워줘.

대상·동작:
- 항목 리스트 아래에 버튼을 배치
- 클릭 시 표준 confirm() 다이얼로그 → "확인" 누른 경우에만 완료된 항목 일괄 삭제
- 삭제 결과는 localStorage 에도 즉시 반영

엣지 케이스:
- 완료된 항목이 0개일 때 버튼은 disabled 상태로 비활성화
- 항목이 추가/완료 토글될 때마다 버튼 활성화 상태 자동 갱신

제약:
- 기존 함수 시그니처 변경 금지
- 외부 라이브러리 도입 금지

검증:
- 어떤 시나리오로 브라우저에서 직접 확인할 수 있는지도 함께 알려줘
```

Claude가 계획을 제시하면 **그대로 받아들이지 마세요.** 다음 중 하나는 꼭 합니다.

- `Ctrl+G` 로 계획을 편집기에서 열어 한 줄이라도 직접 손본다
- 마음에 안 드는 부분 한 가지를 짚어 "X는 Y로 바꿔줘" 라고 지시한다

> 이 단계의 가치는 **'사람이 검토했다'는 흔적을 남기는 것**입니다.

### 5. 구현 → 검증 (6분)

`Shift+Tab` 으로 Normal Mode 복귀.

```
검토한 계획대로 구현해줘.
```

브라우저에서 `index.html` 을 새로고침해 다음을 직접 확인합니다.

```
□ 완료 항목 0개 상태에서 버튼이 비활성화(disabled)되어 있나
□ 항목을 몇 개 완료 처리하면 버튼이 활성화되나
□ 버튼 클릭 시 confirm 다이얼로그가 뜨나
□ "확인" 누르면 완료 항목만 사라지고 미완료 항목은 남나
□ "취소" 누르면 아무것도 변하지 않나
□ 새로고침 후에도 결과가 유지되나
```

문제가 있으면 그 증상을 그대로 복사해 다시 묻습니다.

```
브라우저에서 [증상 그대로]가 발생함.
원인을 찾아 수정하고, 변경 이유도 한 줄 설명해줘.
```

> 잘못된 변경이 끼면: `Esc+Esc` → Rewind 메뉴에서 직전 상태로 되돌립니다.

### 6. 커밋 (2분)

```
변경분을 다음 형식으로 커밋해줘:
[기능] todo — 완료 항목 일괄 삭제 버튼 추가
```

커밋 직전 본인이 `git diff` 한 번 봅니다. **Claude의 변경을 검토 없이 그대로 커밋하는 습관이 가장 위험합니다.**

```bash
git diff HEAD~1
```

문제 없으면 끝.

### 실습 후 점검표

```
□ 빈 폴더에서 시작해 동작하는 TODO 앱이 만들어졌다
□ /init 으로 CLAUDE.md 가 생성되었고, 한 줄 직접 다듬었다
□ Plan Mode 로 4요소(대상·동작·제약·검증)를 갖춘 프롬프트를 한 번 써봤다
□ 브라우저 검증 시나리오를 직접 클릭해서 확인했다
□ 변경분을 git diff 로 직접 보고 커밋했다
□ 한 번이라도 Esc+Esc 또는 /clear 를 사용해봤다
```

여섯 개 중 네 개 이상 체크되면 기초는 충분합니다. 다음은 [심화 가이드](./02_claude-code_심화.md)로 넘어갑니다.

## 10. 트러블슈팅

### Claude가 같은 실수를 반복할 때

```
원인: 잘못된 시도들이 컨텍스트를 오염시킴
해결: 2번 연속 실패 시 → /clear 후 더 구체적인 초기 프롬프트로 재시작
```

### 응답 품질이 점점 저하될 때

```
원인: 컨텍스트 윈도우가 가득 참
해결: /compact 로 압축하거나, /clear 로 초기화 후 핵심 맥락만 다시 제공
```

### 원치 않는 변경이 일어났을 때

```bash
Esc+Esc     # Rewind 메뉴 → 이전 상태로 되돌리기
/rewind     # 동일한 효과
```

> Rewind는 Claude의 작업을 되돌리지만, 이미 실행된 shell 명령은 되돌릴 수 없습니다.

### Claude가 무한히 파일을 탐색할 때

```
원인: 범위가 너무 넓은 요청
해결: @파일명 으로 탐색 범위를 명시적으로 제한
      또는 "OrderService.java와 OrderController.java만 확인해줘" 처럼 범위를 지정
```

### 빌드/테스트 실패 후 Claude가 멈출 때

```
해결: 에러 메시지를 직접 복사해서 "이 에러로 실패함: [에러]. 수정하고 빌드 성공 확인해줘." 로 재지시
```

### 권한 요청이 너무 자주 뜰 때

자주 사용하는 명령어를 `.claude/settings.json`에 미리 허용합니다:

```json
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(./gradlew *)",
      "Bash(npm *)"
    ]
  }
}
```

> 더 자세한 설정 방법은 [심화 가이드](./02_claude-code_심화.md)를 참고하세요.

## 11. 성장 경로

Claude Code 활용 능력은 한 번에 쌓이지 않습니다. 각 단계는 **앞 단계를 충분히 해본 사람만 보이는 한계**를 풀어주는 도구입니다. 자기 위치를 확인하고, 한 칸씩 올라가세요.

### 10단계 사다리

```
[개인 영역 — 본인 작업의 효율화]
  1. 설치 + 일단 써보기 + CLAUDE.md 초기 작성         → ① 기초 §1~§7
  2. SDD/TDD 로 검증 기준 세우기                       → ① 기초 §8 + 외부 SDD/TDD 자료
  3. 메모리·컨텍스트 관리                              → ② 심화 §2
  4. 반복되는 요청을 Skills 로 묶기                    → ② 심화 §4
  5. Hooks 로 자동화                                   → ② 심화 §3 + .claude/guides/hooks.md

[도구 확장 — 한 사람이 다룰 수 있는 폭을 넓힘]
  6. Subagent / 병렬 에이전트                          → ② 심화 §6
  7. MCP·Plugin (외부 시스템 연결, 커스텀 묶음)       → ② 심화 §5, §10

[팀 영역 — 여럿이 공유하는 시스템]
  8. 팀 표준화 (CLAUDE.md·rules·skills·hooks 공유)    → ③ 사용법 §3, §5, §6
  9. 데이터 기반 개선 사이클 (회고 → 분석 → 반영)     → ③ 사용법 §7

[시스템화 — 대화 밖에서도 동작하는 Claude]
 10. CI/CD·비대화형 통합 (claude -p, GitHub Actions)  → ② 심화 §9
```

### 단계 진입 신호

다음 단계로 넘어갈 시점은 "이 질문에 답이 막히는 순간"입니다. 도구를 먼저 도입하면 **왜 그게 필요한지** 모른 채 쓰게 됩니다.

| 단계 전환 | 진입 신호 (이 질문에 막히면 다음 단계로) |
|----------|------------------------------------------|
| 2 → 3 | "검증 기준은 잡혔는데, 같은 결정·관습을 매 세션 다시 설명해야 한다." |
| 3 → 4 | "같은 형태의 프롬프트를 이번 주에 세 번째 쓰는데 매번 길게 적고 있다." |
| 4 → 5 | "특정 작업이 끝나면 매번 같은 후속 작업(알림·린트·테스트)을 손으로 한다." |
| 5 → 6 | "한 작업의 분석 범위가 너무 넓어 컨텍스트가 매번 가득 찬다." |
| 6 → 7 | "Slack·Jira·DB 같은 외부 시스템 정보가 매 작업마다 필요하다." |
| 7 → 8 | "내가 만든 Skills·규칙을 옆자리 동료가 모른다. 따라하는 비용이 크다." |
| 8 → 9 | "팀 규칙은 있는데 어떤 규칙이 효과 있고 어떤 게 잊혀지는지 데이터가 없다." |
| 9 → 10 | "리뷰·검증을 사람이 매번 트리거하는데, PR 단위로 자동화할 수 있을까?" |

### 두 가지 주의

- **단계를 건너뛰지 마세요.** Hooks(5)부터 건드려도 동작은 합니다. 다만 메모리(3)·Skills(4) 가 안 잡힌 상태에서 Hooks 를 도입하면 **왜 그렇게 만들었는지 본인도 잊어버립니다.**
- **단계를 너무 빨리 끝내려 하지 마세요.** 1~2단계에서 한 달 머물러도 됩니다. 도구를 늘리는 것보다, 같은 도구를 깊게 쓰는 게 결과를 더 키웁니다.

### 어떤 작업이 어느 단계에 해당하는지 모르겠다면

- "내 손이 반복하고 있다" → 4단계 (Skills) 검토 시점
- "Claude가 매번 같은 컨텍스트를 잃어버린다" → 3단계 (메모리) 검토 시점
- "팀원이 같은 실수를 반복한다" → 8단계 (팀 표준화) 검토 시점
- "어떤 규칙이 정말 도움이 되는지 모르겠다" → 9단계 (개선 사이클) 검토 시점
